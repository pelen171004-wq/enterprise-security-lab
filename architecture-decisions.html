Architecture Decisions

ADR-style documentation of key design decisions — rationale, alternatives considered, and trade-offs accepted. Each decision is traceable to a specific security or operational requirement.

ADR
ADR-001
FortiGate 60E as primary NGFW over pfSense/OPNsense
ACCEPTED
Decision & Rationale
FortiGate 60E selected as the edge NGFW. Primary driver: enterprise skill transferability. Fortinet is the leading NGFW vendor in enterprise environments (Gartner MQ). Skills built on FortiOS 7.4 — IPS policy authoring, UTM profiles, FortiGuard — transfer directly to production environments. pfSense/OPNsense, while capable, are not commonly found in enterprise security engineering roles.
Alternatives & Trade-offs
✓ Enterprise-grade UTM stack in one device
✓ FortiGuard threat intelligence included
✓ CLI syntax matches production FG environments
✗ Higher cost than open-source alternatives
✗ FortiGuard subscription required for full UTM
ADR-002
Two CBS250-8PP-D in cascade over single larger switch
ACCEPTED
Decision & Rationale
Two 8-port CBS250-8PP-D switches in cascaded topology selected over a single 24-port switch. Rationale: PoE+ budget (65W per switch = 130W total), physical placement flexibility, and the opportunity to document and practice inter-switch trunking — a skill directly relevant to multi-floor enterprise deployments.
Alternatives & Trade-offs
✓ 802.1Q trunk practice between switches
✓ 130W PoE budget for IP phones, APs
✓ Physical redundancy path possible
✗ Cascaded = single point of failure at SW-A
✗ 16 access ports total (vs 22 on 24-port)
ADR-003
VLAN ID allocation: 20/30/40/50/60/70 (not 1/2/3...)
ACCEPTED
Decision & Rationale
VLAN IDs allocated in tens (20, 30, 40...) rather than sequential (1, 2, 3...). Rationale: operational readability — VLAN ID visually communicates zone membership. 10.10.20.x subnet matches VLAN 20. Leaves room for future expansion within each decade. VLAN 999 as native/unused follows enterprise best practice of not using VLAN 1 as native.
Alternatives & Trade-offs
✓ VLAN ID matches 3rd octet of subnet
✓ Easy mental mapping: VLAN20 = 10.10.20.x
✓ VLAN 1 never used as data VLAN
✗ Wastes VLAN IDs 1–19, 21–29 etc.
ADR-004
10.10.x.x addressing over 192.168.x.x
ACCEPTED
Decision & Rationale
10.10.x.x /24 subnets chosen over the more common 192.168.x.x. Rationale: differentiation from home network defaults. Most consumer routers use 192.168.0.x or 192.168.1.x — using 10.10.x.x prevents overlap if lab is connected to ISP router. Also mirrors how enterprise networks commonly use 10.x.x.x space for internal addressing.
Alternatives & Trade-offs
✓ No overlap with typical home/ISP ranges
✓ Mirrors enterprise 10.x.x.x convention
✓ Subnet matches VLAN ID (10.10.20.x = VLAN20)
✗ Slightly less intuitive than 192.168.x.x for beginners
